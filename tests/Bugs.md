# Баг-сценарии хэндлеров — my_ai_bot

Список потенциальных багов, вытекающих из плана тестов `tests/test_handlers.py`.
Каждый сценарий — воспроизводимый кейс с ожидаемым и фактическим поведением.
Формат: **ID / Компонент / Severity / Шаги / Ожидание / Факт (гипотеза) / Как проверить**.

---

## common.py

### BUG-01 — `/start` не сбрасывает `awaiting_input` при рестарте бота
- **Компонент:** `handlers/common.py :: cmd_start`
- **Severity:** High
- **Шаги:**
  1. Пользователь жмёт "Сменить ключ Cerebras" → `awaiting_input = "cerebras_key"`.
  2. Бот перезапускается / пользователь передумал и жмёт `/start`.
- **Ожидание:** `awaiting_input` сброшен в `None`, следующее обычное сообщение уходит в чат, а не в `set_key`.
- **Факт (гипотеза):** если в `cmd_start` условие `not created and user.awaiting_input is not None` реализовано с ошибкой (например, проверяется `created` неверно), ключ пользователя может быть перезаписан произвольным текстом сообщения.
- **Как проверить:** `test_start_resets_awaiting_input_for_existing_user` — создать юзера с `awaiting_input="cerebras_key"`, вызвать `cmd_start`, проверить `user.awaiting_input is None` после `refresh_from_db()`.

### BUG-02 — `/reset` затирает больше, чем историю
- **Компонент:** `handlers/common.py :: cmd_reset`
- **Severity:** Critical
- **Шаги:** Задать ключ и провайдера → вызвать `/reset`.
- **Ожидание:** `history == []`, `current_provider` и `api_keys` не изменились.
- **Факт (гипотеза):** если `clear_history()` в модели по ошибке чистит весь объект (например, пересоздаёт `User` или сбрасывает лишние поля), ключи/провайдер потеряются молча.
- **Как проверить:** `test_reset_only_clears_history` — assert по `current_provider` и `get_key()` до/после.

### BUG-03 — `/help` не синхронизирован с `SUPPORTED_PROVIDERS`
- **Компонент:** `handlers/common.py :: cmd_help`
- **Severity:** Low
- **Шаги:** Добавить нового провайдера в `SUPPORTED_PROVIDERS`, не трогая `PROVIDER_LABELS`/текст помощи.
- **Ожидание:** `/help` автоматически перечисляет актуальный список.
- **Факт (гипотеза):** список захардкожен строкой в другом месте → рассинхрон.
- **Как проверить:** `test_help_lists_all_supported_providers` — assert каждый provider из `SUPPORTED_PROVIDERS` есть в тексте ответа.

## settings.py

### BUG-04 — Переключение на провайдера без ключа проходит успешно
- **Компонент:** `handlers/settings.py :: cb_use_provider`
- **Severity:** Critical
- **Шаги:** Callback `settings:use:cerebras` для юзера без сохранённого ключа.
- **Ожидание:** `callback.answer(show_alert=True)` с текстом про отсутствие ключа, `current_provider` НЕ меняется, `save()` не вызывается.
- **Факт (гипотеза):** проверка `if user.get_key(provider) is None` может не сработать, если `get_key` возвращает `""` вместо `None` при пустом ключе (тип несовпадение) — тогда переключение пройдёт "на пустой ключ".
- **Как проверить:** `test_use_provider_without_key_blocks_switch` + отдельно `test_get_key_empty_string_vs_none`.

### BUG-05 — `settings:use:<unknown>` не проверяется до обращения к БД
- **Компонент:** `cb_use_provider`
- **Severity:** Medium
- **Шаги:** Подменить `callback.data = "settings:use:doesnotexist"`.
- **Ожидание:** алерт "Неизвестный провайдер", никаких запросов к `User`.
- **Факт (гипотеза):** порядок проверок нарушен — сначала лезем в БД (`_get_user`), потом проверяем provider, лишняя нагрузка + возможен побочный эффект (создание юзера) на мусорный callback.
- **Как проверить:** `test_unknown_provider_short_circuits_before_db` — замокать `User.get_or_create` и убедиться, что он не вызывается при неизвестном provider.

### BUG-06 — Гонка `set_awaiting` + одновременный `/cancel`
- **Компонент:** `cb_set_key_prompt` vs `cmd_cancel`
- **Severity:** Medium
- **Шаги:** Пользователь жмёт "Сменить ключ", почти сразу шлёт `/cancel`, затем присылает произвольный текст (не ключ).
- **Ожидание:** после `/cancel` `awaiting_input is None`, следующий текст уходит в обычный чат.
- **Факт (гипотеза):** если между `set_awaiting(f"{provider}_key")` и обработкой `/cancel` нет единой транзакции/лока — при высокой конкурентности возможна потеря обновления (last write wins), пользователь остаётся "подвешен" в режиме ввода ключа.
- **Как проверить:** сложно юнит-тестом; минимум — `test_cancel_after_setkey_prompt_resets_state` последовательно, и пометка "требует интеграционного/нагрузочного теста" для реальной гонки.

### BUG-07 — Кнопки клавиатуры не обновляются после смены ключа
- **Компонент:** `cb_use_provider` / `_settings_keyboard`
- **Severity:** Low
- **Шаги:** Задать ключ через `_save_api_key` (в chat.py), затем открыть `/settings` заново без пересборки клавиатуры.
- **Ожидание:** значок 🔑 вместо ➕ у провайдера с новым ключом.
- **Факт (гипотеза):** `_save_api_key` в chat.py не дергает `cb_set_key_prompt`/не обновляет исходное сообщение с клавиатурой — старое сообщение `/settings` остаётся с ➕, вводит пользователя в заблуждение.
- **Как проверить:** UI-специфичный баг, юнит-тестом не ловится напрямую; можно проверить хотя бы то, что `_settings_text`/`_settings_keyboard` дают корректный результат при повторном вызове `cmd_settings` после сохранения ключа.

## chat.py

### BUG-08 — `handle_message` обрабатывает ключ, даже если провайдер из `awaiting_input` больше не поддерживается
- **Компонент:** `chat.py :: _save_api_key`
- **Severity:** Medium
- **Шаги:** `awaiting_input = "removedprovider_key"` (провайдер убрали из `SUPPORTED_PROVIDERS` после релиза), пользователь присылает текст.
- **Ожидание:** `set_awaiting(None)`, сообщение об ошибке синхронизации, `set_key` НЕ вызывается.
- **Факт (гипотеза):** проверка `provider not in SUPPORTED_PROVIDERS` есть, но нужно убедиться, что `removesuffix("_key")` корректно режет суффикс даже если в имени провайдера есть `"_key"` как часть (маловероятно, но граничный кейс) — иначе `provider` останется с "хвостом" и ложно не совпадёт со списком, хотя реально совпадает.
- **Как проверить:** `test_save_api_key_unknown_provider_from_awaiting`.

### BUG-09 — Ключ сохраняется "как есть" без валидации формата
- **Компонент:** `_save_api_key`
- **Severity:** Medium
- **Шаги:** Отправить в качестве ключа пустую строку из одних пробелов `"   "`.
- **Ожидание:** либо явный отказ ("ключ не может быть пустым"), либо осознанное решение, что пустая строка = валидный ключ (тогда описать в докстринге).
- **Факт (гипотеза):** `raw_key = message.text.strip()` даст `""`, `set_key(provider, "")` сохранит пустую строку как "ключ задан" (🔑 в UI), хотя фактически ключа нет → пользователь думает что всё настроено, а чат будет падать с `ProviderError` при первом же запросе.
- **Как проверить:** `test_save_api_key_rejects_blank_string` — assert либо на сообщение об ошибке, либо явно задокументировать текущее поведение как caveat.

### BUG-10 — История дополняется сообщением юзера даже при недоступном провайдере
- **Компонент:** `_handle_chat_message`
- **Severity:** Low/Medium
- **Шаги:** `current_provider` отсутствует в `PROVIDERS` (рассинхрон БД/кода).
- **Ожидание:** `add_message` НЕ вызывается вообще (сообщение об ошибке до записи в историю) — судя по коду, `return` идёт раньше `add_message`, так что здесь баг маловероятен, но стоит закрепить тестом как регрессионный барьер.
- **Факт (гипотеза):** если кто-то в будущем переставит порядок строк (сначала `add_message`, потом проверка `ask is None`), история "засорится" сообщениями без ответа.
- **Как проверить:** `test_missing_provider_does_not_touch_history` — явный regression-тест.

### BUG-11 — `add_message("assistant", answer)` вызывается при `ProviderError`
- **Компонент:** `_handle_chat_message`
- **Severity:** High
- **Шаги:** `ask()` бросает `ProviderError("rate limit")`.
- **Ожидание:** в историю ничего не добавляется от ассистента (только `thinking.edit_text` с текстом ошибки), иначе в следующем запросе к провайдеру полетит "фейковый" ответ ассистента вида "⚠️ rate limit" как часть контекста диалога.
- **Факт (гипотеза):** по текущему коду `add_message("assistant", answer)` стоит ПОСЛЕ try/except и после `return` в except-ветках, так что баг не должен воспроизводиться — но это критичный инвариант, который легко сломать при рефакторинге (например, кто-то обернёт всё в `finally`).
- **Как проверить:** `test_provider_error_does_not_pollute_history` — после ошибки `user.history` не содержит записи с role="assistant" за этот тик.

### BUG-12 — "💭 Думаю..." остаётся навсегда при таймауте без исключения
- **Компонент:** `_handle_chat_message`
- **Severity:** Medium
- **Шаги:** `ask()` зависает (например, сетевой SDK без своего таймаута) дольше, чем Telegram ждёт ответа на callback/webhook.
- **Ожидание:** пользователь либо получает ошибку по таймауту, либо есть внешний таймаут-враппер вокруг `ask`.
- **Факт (гипотеза):** в коде нет `asyncio.wait_for` / таймаута вокруг `await ask(...)` — если конкретный провайдер зависнет без исключения, сообщение "💭 Думаю..." останется навечно, а хэндлер (и, возможно, весь update-обработчик) будет висеть.
- **Как проверить:** юнит-тестом не ловится (нужен интеграционный/нагрузочный тест с `asyncio.sleep` в фейковом провайдере + `pytest.mark.timeout`), но стоит завести как тех.долг.

### BUG-13 — Не непойманное исключение всё равно попадает в лог с полным трейсом, а не только в `logger.exception`
- **Компонент:** `_handle_chat_message`, общий `except Exception`
- **Severity:** Low
- **Шаги:** `ask()` бросает произвольное исключение SDK (не `ProviderError`).
- **Ожидание:** пользователю — общий текст "⚠️ Непредвиденная ошибка...", в логах — полный traceback через `logger.exception`.
- **Факт (гипотеза):** нужно убедиться, что except-блок не "проглатывает" исключение молча в проде (например, если логгер не сконфигурирован в `main.py` — сообщение просто теряется). Это не баг хэндлера, но риск конфигурации.
- **Как проверить:** `test_unexpected_exception_is_logged` через `caplog.at_level(logging.ERROR)`.

### BUG-14 — Фильтр `F.text & ~F.text.startswith("/")` пропускает "похожие на команду" сообщения
- **Компонент:** `chat.py`, регистрация роутера
- **Severity:** Low
- **Шаги:** Пользователь пишет `"/nonexistent_command"` или просто текст, начинающийся с `/`, но не являющийся зарегистрированной командой (например `"/ Привет"` с пробелом, или сообщение вида `"/"`).
- **Ожидание:** явное поведение — либо "неизвестная команда", либо осознанно игнорируется.
- **Факт (гипотеза):** сейчас такие сообщения молча проваливаются мимо `chat.py` (из-за `~F.text.startswith("/")`) и мимо всех известных команд → пользователь не получает вообще никакого ответа, выглядит как "бот завис".
- **Как проверить:** `test_unknown_slash_command_gives_some_response` — на уровне интеграции с полным `Dispatcher`, а не одного хэндлера.

## Сквозные (кросс-хэндлерные) сценарии

### BUG-15 — Порядок регистрации роутеров важен, но ничем не гарантирован
- **Компонент:** `main.py` (не показан) + весь `awaiting_input`-флоу
- **Severity:** Critical
- **Шаги:** Кто-то в `main.py` регистрирует `chat.router` раньше `settings.router`.
- **Ожидание:** независимо от порядка регистрации, ввод ключа должен обрабатываться корректно, т.к. `chat.py` сам явно проверяет `user.awaiting_input` первой строкой.
- **Факт (гипотеза):** README/докстринги говорят "либо порядок роутеров, либо явная проверка в chat.py" — то есть защита состояния зависит от того, что оба условия держатся в голове разработчика одновременно. Если кто-то уберёт проверку `awaiting_input` из `chat.py`, посчитав её "дублирующей", логика сломается тихо, без ошибки на этапе разработки.
- **Как проверить:** `test_chat_handler_checks_awaiting_input_regardless_of_router_order` — юнит-тест уже покрывает это на уровне функции `handle_message`, но стоит явно задокументировать инвариант в самом тесте (docstring / комментарий "не убирать эту проверку").

### BUG-16 — Провайдер существует в `SUPPORTED_PROVIDERS`, но не в `PROVIDERS`
- **Компонент:** `settings.py` (даёт выбрать) + `chat.py` (не находит в `PROVIDERS`)
- **Severity:** High
- **Шаги:** Добавили нового провайдера в `SUPPORTED_PROVIDERS` (модели/БД), но забыли добавить функцию + строку в словарь `PROVIDERS` в `providers.py`.
- **Ожидание:** пользователь либо не может выбрать такого провайдера в `/settings`, либо получает понятную ошибку при первом сообщении.
- **Факт (гипотеза):** `settings.py` разрешит выбрать провайдера (проверяется только `SUPPORTED_PROVIDERS` и наличие ключа), а `chat.py` уже после переключения скажет "Провайдер «X» не найден" — пользователь окажется в состоянии "выбрал модель, но чат не работает", без возможности понять, что делать, кроме как снова открыть `/settings`.
- **Как проверить:** `test_supported_providers_and_providers_dict_are_in_sync` — прямой assert `set(SUPPORTED_PROVIDERS) == set(PROVIDERS.keys())`, чтобы ловить рассинхрон на этапе CI, а не в проде.
