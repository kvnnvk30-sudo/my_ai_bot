# Чек-лист тестирования — my_ai_bot (без common/settings/chat.py, они уже покрыты багрепортом)

Проверено по факту присланного кода (`main.py`, `config.py`, `providers.py`, `database/models.py`,
плюс беглый взгляд на `handlers/*` — не для повторного тестирования, а чтобы сверить с тем,
что реально импортируется/вызывается оттуда).

---

## 1. `providers.py`

- [ ] `ask_cerebras` вызывает `client.chat.completions.create(model="gpt-oss-120b", messages=[...])` с системным промптом первым сообщением (`SYSTEM_PROMPT`) + всей `history` без изменений порядка
- [ ] `_require_key` реально выполняется **до** обращения к SDK (пустой/`None` ключ → `ProviderError`, `Cerebras(api_key=...)` не создаётся)
- [ ] Любое исключение из `_ask_cerebras_sync` (сеть, 401/403/429/5xx, `ImportError` самого SDK) оборачивается в `ProviderError(f"Cerebras: {e}")` — хэндлер `chat.py` ловит только один класс
- [ ] `asyncio.to_thread` действительно не блокирует event loop (синхронный SDK-вызов не морозит остальных пользователей бота) — актуально в связке с нагрузочным тестом
- [ ] Ответ SDK (`response.choices[0].message.content`) — что будет, если структура ответа неожиданно другая (`choices` пустой, `content is None`) — сейчас упадёт `IndexError`/`AttributeError`, попадёт в `except Exception` → обычный `ProviderError`, но с обезличенным текстом
- [ ] Сигнатура `ask_cerebras(history, api_key) -> str` соответствует тому, как её вызывает `chat.py`: `await ask(user.history, api_key)` — совпадает

## 2. `database/models.py`

- [ ] `set_key`/`get_key`/`clear_key` кидают `ValueError` на неизвестном провайдере — это происходит **до** похода в БД (соответствует принципу из settings.py, но здесь это отдельная защита на уровне модели — стоит проверить, что она не даёт «второй шанс» пройти мимо проверки в хэндлере)
- [ ] `_load_keys_dict`: битый/старый `api_keys_encrypted` (`InvalidToken`, невалидный JSON) не роняет бота — тихо возвращает `{}`. Стоит явно протестировать оба случая по отдельности
- [ ] `_get_fernet()` закэширован через `lru_cache(maxsize=1)` — **если `SECRET_KEY` в `.env` невалиден (не 32-байтный base64), это не всплывёт при старте бота**, а только при первом реальном вызове `set_key`/`get_key` (первый пользователь, зашедший в `/settings` или начавший чат). Нужен отдельный тест на старте/smoke-проверка валидности ключа сразу при запуске
- [ ] `add_message` обрезает `history` до `HISTORY_LIMIT = 20` — проверить обрезание именно с начала (старые сообщения улетают, а не новые)
- [ ] Конкурентные `add_message`/`set_key` от одного пользователя (два параллельных апдейта) — `self.history.append(...)` в памяти объекта + `save(update_fields=["history"])`: два параллельных инстанса `User` для одного `telegram_id`, полученные через два разных `get_or_create`, перезапишут историю друг друга (last write wins) — стоит явно протестировать этот кейс, это не мифический риск, а видно прямо в коде
- [ ] Значения по умолчанию новой записи: `current_provider="cerebras"`, `history=[]`, `awaiting_input=None` — корректны при первом `get_or_create`

## 3. `config.py` / `.env.example`

- [ ] `Settings()` создаётся на уровне модуля (`settings = Settings()` — исполняется при импорте `src.config`). Отсутствие `BOT_TOKEN` или `SECRET_KEY` в `.env` → `pydantic.ValidationError` **прямо при импорте**, ещё до `main()` — проверить, что сообщение об ошибке понятное, а не голый traceback
- [ ] `DB_URL` по умолчанию `sqlite://db.sqlite3` — проверить, что это осознанный дефолт для prod, а не забытая dev-заглушка (в тест-плане это помечено как риск «секреты/режимы»)
- [ ] В `config.py` **нет** явного разделения dev/prod (нет поля вроде `ENV`/`DEBUG`) — если это разделение существует, то не в этом файле; стоит либо подтвердить, что оно не нужно, либо завести баг «дев/прод режимы не разделены на уровне конфига», раз тест-план это требует (раздел 3, «конфигурационное тестирование»)
- [ ] `.env.example` не проверялся на попадание реальных секретов (кода нет в присланных файлах) — нужно свериться отдельно

## 4. `main.py`

- [ ] `init_db()` вызывается и **await**-ится до создания `Bot`/`Dispatcher` и до `start_polling` — по коду порядок правильный, гонки нет
- [ ] Порядок `dp.include_router(module.router)` — `common, settings_handlers, chat`. Здесь важно не только для `awaiting_input` (это чек `chat.py`, уже закрыт BUG-15), а для **дубликата `/cancel`** (см. п.5 ниже) — порядок регистрации решает, чей `/cancel` реально сработает
- [ ] `finally: bot.session.close() + Tortoise.close_connections()` — отработает ли корректно при `KeyboardInterrupt`/`SIGTERM` во время `start_polling` (грейсфул-шатдаун), не только при штатном исключении
- [ ] `logging.basicConfig(level=logging.INFO)` — единственная настройка логгера на всё приложение; хвост из BUG-13 («конфигурация логгера не проверялась») закрывается именно здесь: проверить, что уровня `INFO` достаточно и `logger.exception` в `chat.py` реально долетает до вывода

## 5. Найдено при чтении кода — новое, не было в предыдущей версии чек-листа

- [ ] **Дублирующийся `/cancel`**: команда `Command("cancel")` зарегистрирована и в `handlers/common.py`, и в `handlers/settings.py`, с почти идентичной логикой. Т.к. в `main.py` порядок `common → settings_handlers → chat`, обработчик из `common.py` перехватывает апдейт первым, а версия в `settings.py` — мёртвый код (никогда не вызывается через `router`). Нужно решить: убрать дубликат или объяснить, зачем он оставлен
- [ ] В `settings.py` строка `async def cmd_cancel(message: Message) -> None:#` — висящий `#` сразу после `:` (не баг, но грязный код/возможный артефакт правки, стоит почистить)
- [ ] **Дыра в фильтре `chat.py`**: `@router.message(F.text & ~F.text.startswith("/"))` — если пользователь в состоянии `awaiting_input` (ждём API-ключ) пришлёт строку, начинающуюся с `"/"` (в т.ч. случайно, если ключ по формату так выглядит), сообщение не попадёт вообще ни в один хэндлер — бот промолчит, а `awaiting_input` останется висеть. Это отдельный, более узкий кейс, чем BUG-14 (там про неизвестные команды), — стоит завести отдельный тест/баг

- [ ] `README.md` соответствует фактическому списку провайдеров и командам бота (раздел 7 риска — «устаревшая документация»)

## 7. Нефункциональные виды тестирования (раздел 3 тест-плана — пока не выполнялись)

### Acceptance (ручной прогон в реальном Telegram)
- [ ] `/start` от нового пользователя — приветствие + чистое состояние
- [ ] `/reset` не трогает ключи и текущего провайдера, только историю
- [ ] Смена провайдера без ключа → бот блокирует и просит ключ
- [ ] Ввод ключа → успешное сохранение → индикатор ключа в клавиатуре появляется
- [ ] Обычный диалог с Cerebras (реальный вызов) — ответ приходит, история копится
- [ ] Отправка `/nonexistent_command` — сверить с BUG-14 (ожидаемо: тишина)
- [ ] Отправка сообщения во время «зависшего» ответа провайдера (см. BUG-12) — что видит пользователь

### Нагрузочное
- [ ] Параллельные сообщения от нескольких синтетических пользователей — нет утечки состояния между пользователями
- [ ] Параллельные сообщения от **одного** пользователя подряд — история не дублируется/не бьётся

### Конфигурационное
- [ ] Запуск в dev-режиме (docker-compose) — smoke-check
- [ ] Запуск в prod-режиме — smoke-check
- [ ] Запуск без `.env` вовсе (только пример) — ожидаемое поведение (ошибка на старте, а не в рантайме)

### Локализационное
- [ ] Все RU-тексты ответов и ошибок читаются нормально (нет технических артефактов, `None`, `{}` в тексте)
- [ ] Тексты ошибок провайдера понятны пользователю (не «сырой» stacktrace)
