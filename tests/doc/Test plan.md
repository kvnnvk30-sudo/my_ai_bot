# Тест-план — my_ai_bot (Telegram-бот, провайдер Cerebras gpt-oss-120b)

**Версия документа:** 2.0
**Проект:** my_ai_bot
**Назначение документа:** основа для проведения тестирования — что и как тестировать. Результаты прогонов, статусы багов и итоги в этот документ не входят; они фиксируются отдельно (баг-репорты, отчёт о прогоне).

---

## 1. Цели тестирования

Проверить, что Telegram-бот с AI-провайдером Cerebras (gpt-oss-120b):
1. Корректно управляет состоянием пользователя (текущий провайдер, зашифрованные ключи, история диалога, режим ожидания ввода).
2. Не теряет и не портит данные пользователя при штатных операциях (`/reset`, `/start`, смена ключа).
3. Не даёт обойти бизнес-правила (например, переключиться на провайдера без ключа).
4. Устойчиво ведёт себя при ошибках провайдера, сетевых зависаниях и нераспознанных командах.
5. Не рассинхронизируется между слоями (конфигурация провайдеров в БД и в диспетчере).

## 2. Объект тестирования (Scope)

| Модуль | Файл |
|---|---|
| Хэндлеры | `src/handlers/common.py` |
| Хэндлеры | `src/handlers/settings.py` |
| Хэндлеры | `src/handlers/chat.py` |
| Провайдер | `src/providers.py` |
| Модель данных | `src/database/models.py` |
| Конфигурация | `src/config.py`, `.env.example` |
| Точка входа | `src/main.py` (регистрация роутеров) |

### Вне скоупа
- Прод-нагрузка / реальные пользователи (только синтетическая имитация).
- Инфраструктура деплоя (Docker) — только smoke-проверка запуска.

## 3. Виды тестирования и подход

| Вид | Что проверяем | Метод |
|---|---|---|
| Функциональное (unit/компонентное) | Логика хэндлеров, модели, провайдера в изоляции | `pytest` + `pytest-asyncio` + `AsyncMock`, in-memory SQLite |
| Интеграционное | Взаимодействие хэндлеров через реальный `aiogram.Dispatcher` | `dp.feed_update()` с замоканным `bot` |
| Acceptance (приёмочное) | Сценарии глазами пользователя в реальном Telegram | Ручное прохождение чек-листа (раздел 6) |
| Нагрузочное | Поведение при параллельных сообщениях от нескольких пользователей | Синтетический скрипт с параллельными вызовами хэндлера |
| Конфигурационное | `.env`, режимы dev/prod, docker-compose | Ручная проверка запуска в каждом режиме |
| Локализационное | RU-тексты хэндлеров и ошибок | Ручной просмотр текстов ответов |

## 4. Тестовое окружение

- Python 3.11+, зависимости из `requirements.txt`.
- Тестовая БД: SQLite in-memory (изолированно от прод-файла `db.sqlite3`).
- Отдельный тестовый Telegram-бот (не прод-токен) для acceptance-сценариев.
- Тестовый API-ключ Cerebras для сценариев с реальным вызовом провайдера.

## 5. Критерии входа / выхода

**Вход (можно начинать тестирование):**
- Код смержен в целевую ветку.
- Тестовое окружение (раздел 4) поднято и доступно.

**Выход (можно закрывать цикл тестирования):**
- Все пункты чек-листа (раздел 6) пройдены.
- Автотесты по каждому модулю из раздела 2 написаны и прогнаны хотя бы один раз.
- Открытые баги классифицированы по критичности, и по каждому принято явное решение (фикс / отложено с обоснованием).

## 6. Объекты тестирования по компонентам

Ниже — что именно должно быть покрыто тестами в каждом компоненте, на уровне аспектов, а не конкретных сценариев. Конкретные тест-кейсы и чек-листы ведутся отдельно.

| Компонент | Аспекты, подлежащие проверке |
|---|---|
| `common.py` | Корректность сброса состояния пользователя (`/start`, `/reset`, `/cancel`); `/reset` не должен затрагивать ничего, кроме истории; актуальность списка провайдеров в `/help`. |
| `settings.py` | Бизнес-правило «нет ключа → нет переключения»; поведение при пустом/некорректном ключе; обработка неизвестного провайдера до обращения к БД; консистентность клавиатуры с фактическим состоянием ключей. |
| `chat.py` | Обработка обычных сообщений; поведение при ошибке провайдера (`ProviderError`) и при непредвиденном исключении; поведение при зависании вызова провайдера; реакция на нераспознанные команды; целостность истории при любом из этих сценариев. |
| `providers.py` | Корректность вызова SDK провайдера; обработка сетевых и API-ошибок; соответствие сигнатуры функции интерфейсу, ожидаемому диспетчером `PROVIDERS`. |
| `database/models.py` | Шифрование/дешифрование ключей (`set_key`/`get_key`); целостность полей `history`, `awaiting_input`, `current_provider` при конкурентных операциях; валидация длины/типа полей. |
| `config.py`, `.env.example` | Загрузка конфигурации, разделение режимов dev/prod, отсутствие секретов в репозитории. |
| `main.py` | Регистрация роутеров и её независимость от порядка; инициализация ORM до старта поллинга/вебхука. |
| Сквозное | Синхронность `SUPPORTED_PROVIDERS` (БД) и `PROVIDERS` (диспетчер). |

## 7. Категории рисков

- Потеря пользовательских данных при штатных командах (`/reset` и подобные).
- Обход бизнес-правил через рассинхрон типов (`None` vs `""`, отсутствующие ключи в словарях).
- Тихие сбои — бот не отвечает вообще, вместо явной ошибки.
- Рассинхрон конфигурации между слоями (БД / диспетчер / документация).
- Устаревшая документация (`README.md`), которая может не совпадать с фактическим составом провайдеров в коде — сверять перед каждым релизом.

## 8. Роли

Для команды из одного разработчика роли автора кода, автора тестов и того, кто проходит acceptance-чек-лист, лучше разносить по времени (не сразу после написания кода), чтобы не пропускать очевидное.

## 9. Артефакты

- Тесты по каждому модулю (unit/интеграционные).
- Отчёт о прогоне — отдельный документ, фиксирующий факт: что реально прогонялось, что нет, какие баги подтверждены/не подтверждены. Не часть этого плана.
- Чек-листы сценариев (acceptance и др.) — ведутся отдельно от этого документа.
